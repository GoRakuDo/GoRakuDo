# AnkiDroid Performance Plan

This document lists the main areas to optimize DenChou for AnkiDroid. The goal is not to remove the visual identity of the template, but to stop the reviewer from loading and preparing features that are not needed on every card.

## Goal

Make DenChou feel lighter on Android while keeping the same core experience:

- fast card open
- smooth answer reveal
- fewer network requests
- less JavaScript work during normal review
- less GPU-heavy styling on low-end devices

## Current Evidence

The current template has several heavy pieces that run or load during normal card review.

| Area | Evidence | Why it matters on AnkiDroid |
|---|---|---|
| Main stylesheet | `Template/styling.css` is about 188 KB | CSS parsing happens on every card render. Large CSS also increases style recalculation cost. |
| Settings script | `_denchou_settings_v0.1.0.js` is about 97 KB | Preferences are rarely opened during review, but the script is currently prepared from the card side. |
| Back template | `back.html` is about 82 KB | Large inline logic makes the answer side heavier before deferred scripts even run. |
| Settings stylesheet | `_denchou_settings_v0.1.0.css` is about 42 KB | Modal-only CSS should not matter during normal reviewing. |
| Deferred features | `_denchou_deferred_v0.1.0.js` is about 23 KB | Image/lightbox/Jitendex hover behavior adds listeners and DOM work. |
| Kanji hover | `_denchou_kanji_hover_v0.1.0.js` is about 18 KB | Desktop-oriented hover features have limited value on touch-first Android. |
| Font loading | `styling.css` imports six Gen Interface JP CDN weight files | Multiple network CSS/font requests can slow first render on mobile. |
| Font Awesome | `front.html` and `back.html` load Font Awesome from CDN | A full icon library is loaded for a small number of icons. |

## Priority A — Highest Impact, Lowest Visual Risk

### 1. Lazy-load Preferences only when the gear button is tapped

**Current state:**

Both `front.html` and `back.html` dynamically load `_denchou_settings_v0.1.0.js` with a `<script>` tag in their deferred setup. If the script is already present on `window` for the current card, they call `denchouBuildMenu()` instead.

**Problem:**

The settings panel is a tool. Normal review does not need it. Loading a 97 KB settings script on every card is like carrying the whole toolbox just to read a flashcard.

**Plan:**

- Keep the gear button visible.
- Do not load `_denchou_settings_v0.1.0.js` during card initialization.
- On first gear tap:
  - inject `_denchou_settings_v0.1.0.css` if needed
  - inject `_denchou_settings_v0.1.0.js`
  - open the menu after the script is ready
- Cache the loaded state in `window` so the second open within the same card is instant.

Note: `window` is recreated between cards. The main win is not cross-card caching; the win is avoiding settings script parsing on cards where the user never opens Preferences.

**Expected result:**

Normal review becomes lighter without changing the UI.

### 2. Replace Font Awesome with Lucide inline SVG icons

**Current state:**

Both `front.html` and `back.html` load:

```html
https://cdnjs.cloudflare.com/ajax/libs/font-awesome/6.0.0-beta3/css/all.min.css
```

**Problem:**

DenChou only needs a limited icon set. Loading the full Font Awesome stylesheet and icon font for this is expensive on mobile.

Use Lucide as the replacement source, but do **not** load the Lucide runtime package in Anki. Copy only the exact raw SVG shapes needed by the template and inline them directly in the HTML/JS that renders the controls.

Lucide icon candidates verified from the official icon pages:

- `sun`
- `moon`
- `settings`
- `lightbulb`
- `circle-question-mark`
- `image`
- `copy`
- `x`
- `columns-2`
- `chevron-down`
- `chevron-up`

Suggested Font Awesome to Lucide mapping:

| Current Font Awesome | Suggested Lucide | Note |
|---|---|---|
| `fa-sun` | `sun` | Dark mode toggle. |
| `fa-moon` | `moon` | Dark mode toggle. |
| `fa-cog` | `cog` or `settings` | `cog` is closer to the old Font Awesome gear; `settings` is acceptable if a slight visual refresh is desired. |
| `fa-lightbulb` | `lightbulb` | Audio/hint cue. |
| `fa-question` | `circle-question-mark` | Hint/help cue. Lucide's version is circled, while Font Awesome's `fa-question` is bare. |
| `fa-image` | `image` | Image toggle. |
| `fa-clone` | `copy` | Closest match for the overlapping-squares clone icon. |
| `fa-times` | `x` | Close button. |
| `fa-chevron-down` | `chevron-down` | Expand/collapse arrow. |
| `fa-chevron-up` | `chevron-up` | Expand/collapse arrow, if needed. |
| `fa-columns` | `columns-2` | Settings/layout icon, if needed. |

**Plan:**

- Replace `<i class="fas ...">` icons with inline Lucide SVG.
- Remove the Font Awesome CDN link from both templates.
- Keep Lucide as copied SVG markup only; do not add a Lucide JS package, icon font, or CDN runtime.
- Keep tap targets and accessibility labels clear.
- Use `stroke="currentColor"` so icons inherit DenChou colors naturally.
- Use `aria-hidden="true"` for purely decorative icons, or add a button-level `aria-label` for icon-only controls.
- Add Lucide (ISC) and Feather (MIT) license notices to the release checklist or third-party notices before publishing.

**Expected result:**

Fewer network requests, no icon font loading, and less CSS/font parsing.

### 3. Reduce Gen Interface JP CDN weights for the mobile-first build

**Current state:**

`styling.css` imports six Gen Interface JP CDN files:

```css
400.css
500.css
600.css
display-600.css
display-700.css
display-800.css
```

**Problem:**

This is visually rich, but heavy for AnkiDroid. Most cards can look good with fewer weights.

Important caveat: these CDN imports are also the documented AnkiDroid fallback when local `collection.media` font files are missing. Reducing the CDN list is only safe if the release guarantees the matching local `.woff2` files are shipped and synced, or if the missing weights are intentionally allowed to fall back to system fonts.

**Plan:**

- Keep the full six-weight stack for the normal build and for CDN fallback safety.
- Create an optional lightweight AnkiDroid profile only when local font files are guaranteed, using:
  - regular 400
  - semibold 600
- Avoid display weights unless a card actually needs them.

**Expected result:**

Lower first-load cost and fewer font fallback delays.

## Priority B — Good Gains, Slightly More Care Needed

### 4. Initialize Lightbox only after the first image tap

**Current state:**

`_denchou_deferred_v0.1.0.js` prepares image switching, lightbox controls, keyboard shortcuts, wheel zoom, mouse pan, touch pan, and pinch zoom during setup.

**Problem:**

Cards with images get many event listeners even if the user never opens the lightbox. On AnkiDroid, touch and layout calculations can be expensive.

**Plan:**

- Keep basic image tap detection.
- Move full lightbox setup behind the first image tap.
- After setup, reuse the same lightbox instance for that card.
- Keep grid view and pinch zoom, but only after the user asks for the lightbox.

**Expected result:**

Cards with images open faster. Heavy behavior only exists when used.

---

### 5. Reduce deck preset fetch cache busting

**Current state:**

`front.html` fetches preset files with `?t=${Date.now()}` when the deck changes.

**Problem:**

`Date.now()` forces a fresh request and bypasses caching. This is useful during development, but not ideal for mobile review.

**Plan:**

- Replace timestamp cache busting with a stable version query, such as `?v=0.1.0`, only after a refresh strategy is defined.
- Refresh cache when template version changes, when the user explicitly saves presets, or when synced preset content may have changed on another device.
- Possible strategies:
  - bump the query version on every preset write
  - store a preset content version in the cache
  - keep `Date.now()` only for explicit refresh/cache-miss paths

**Expected result:**

Less repeated media fetching during review.

---

### 6. Make Jitendex hover behavior optional on mobile

**Current state:**

`_denchou_deferred_v0.1.0.js` is loaded on mobile and desktop. Inside it, `jitendexHover()` scans glossary elements and attaches hover/click behavior. This is separate from `_denchou_kanji_hover_v0.1.0.js`, which is already gated off on mobile.

**Problem:**

Hover-first interactions are less natural on Android. Some of this work can be disabled or simplified for touch devices.

**Plan:**

- Add a mobile gate around the `jitendexHover()` call inside `_denchou_deferred_v0.1.0.js`.
- Keep essential tap-to-expand behavior.
- Skip purely hover-based opacity transitions on AnkiDroid.

**Expected result:**

Less DOM scanning and fewer listeners on mobile cards.

## Priority C — Visual Tuning and CSS Diet

### 7. Add a lightweight visual preset

**Current state:**

DenChou uses transitions, shadows, blur, rounded panels, and animated reveals.

**Problem:**

Blur and large shadows can hit mobile GPU harder than expected, especially on older Android devices.

**Plan:**

Create an **AnkiDroid Lite** preset: a reduced-motion, lighter-shadow visual profile for Android review. It should reduce:

- large `box-shadow`
- heavy `filter: blur(...)`
- long transitions
- decorative animation

Keep:

- OKLCH colors
- readable spacing
- DenChou typography
- core card layout

Note: OKLCH support depends on the Android System WebView version. Keep OKLCH as the DenChou standard, but test the Lite preset on the target AnkiDroid/WebView version before release.

**Implementation:**

- `--ankidroid-lite: false` CSS variable controls the mode.
- `--ankidroid-auto-optimize: true` automatically applies lite mode when `window.innerWidth ≤ 600px`.
- Preferences toggle: Card Behavior → General Settings → Auto Optimize.
- Resize listener updates dynamically.

**Expected result:**

Users can choose speed without losing the DenChou feel.

---

### 8. Split modal-only CSS from always-loaded CSS

**Current state:**

`_denchou_settings_v0.1.0.css` is only useful when Preferences is open.

**Problem:**

Modal-only styles should not affect the normal review path.

**Plan:**

- Load settings CSS only with the settings panel.
- Keep card-critical CSS in `styling.css`.
- Avoid moving core layout variables out of `styling.css` unless they are guaranteed to be loaded before card render.

**Expected result:**

Cleaner normal review path and less CSS parsing.

## Verification Checklist

After each step, test on both Anki Desktop and AnkiDroid.

- Card opens without console errors.
- Front side renders correctly.
- Back side renders correctly.
- Audio buttons still work.
- Dark mode toggle still works.
- Preferences still opens when tapped.
- Save to Anki still works on desktop.
- Images still open in lightbox.
- Pinch zoom still works on Android.
- Deck presets still apply.
- Offline mode still falls back to local/system fonts.
- Kanji hover tooltip still works on desktop.
- Settings panel still opens and renders correctly on AnkiDroid Lite.
- Auto Optimize toggle works (ON: lite mode on small screens, OFF: no auto).
- Lite mode reduces shadows and blur correctly on AnkiDroid.

## Non-goals

Do not do these in the first optimization pass:

- rewrite the whole template architecture
- remove DenChou's visual identity
- remove OKLCH color usage
- drop Anki Desktop behavior to optimize Android only
- minify files before the code is stable and reviewed

## Recommended First Step

Start with lazy-loading Preferences.

This is the best first step because it removes the largest unnecessary JavaScript cost from normal review while keeping the card design almost unchanged.
